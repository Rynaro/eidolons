## What and why

ECM (Eidolons Context Management) is the fourth sibling spec alongside
EIIS/ECL/ESL: it governs the context economy of a running Eidolons session —
metering utilization, deciding when to externalize state to memory, when to
compact, and when to abandon the window for a fresh handoff — **mechanically,
with no human present to notice degradation** (`docs/specs/ecm/spec.md` §1.1–1.2).
The design (P0 of the roadmap) was already drafted pre-campaign at
`docs/specs/ecm/spec.md` + `policy.yaml`, evidenced by
`research/context-lifecycle-survey-2026-07.md` (tri-cycle, S/A-tier anchored
survey of context-rot literature).

This campaign is **P1**: the kernel implementation of that spec — five
mechanisms (meter, pin set, externalize-before-compact, handoff brief, cache
discipline) as bash 3.2 + jq nexus verbs, Claude Code hook recipes, and a
CRYSTALIUM persistence convention — plus the upstream crystalium change it
turned out to require.

### The verify-fail → upstream attribution arc

The change's first end-to-end verification attempt **failed** — both legs of
`eidolons canary --context-handoff` failed at the `crystalium_ingest` step
against the live pinned crystalium 1.7.0 image
(`verify-fail-upstream.md`). This was investigated and attributed, **not**
treated as a maker defect:

1. A structural probe of the pinned image
   (`ghcr.io/rynaro/crystalium@sha256:b6978817…`, v1.7.0) confirmed the
   one-shot CLI verb set is `canary commit doctor dream export forget index
   promote quarantine recall serve` — `ingest` exists only on the MCP tool
   surface (9 tools), producing `Error: No such command 'ingest'` when the
   kernel called it out-of-session.
2. The maker's code (`lib_context.sh:context_try_ingest`) behaved correctly
   under this gap: it fails open to the file floor, never fabricates a
   success, and never silently falls back to `commit` — honoring the AC-16
   frozen constraint that the canonical persist path has no commit-fallback
   branch.
3. The gap had precedent: the same shape as an earlier harness campaign gap
   (`DOSSIER-HARNESS GAP-2`) where out-of-session `recall` shipped in
   crystalium 1.4.0 and out-of-session `commit` shipped in 1.7.0 — `ingest`
   was simply the missing third verb of that pairing.
4. Disposition: the change held at `in_progress` rather than descoping.
   Criteria stayed frozen (SHA `6e0b9b10…`) — 15 of 17 ACs were
   checker-verifiable immediately; AC-4/AC-9 were marked live-verify-blocked
   pending the crystalium 1.8.0 upstream fix, which then landed as its own
   ESL change (see Implementation summary above).

### End-to-end canary — after the fix

`eidolons canary --context-handoff` against a local `crystalium:1.8.0-dev`
image: **PASS on both legs out-of-the-box** — AC-4 (token round-trip) and
AC-9 (tool-origin/quarantined brief still surfaces on default recall).

The AC-9 result closes the risk FORGE's D5 flagged as the single highest-risk
open unknown in its whole deliberation set (`deliberation.md` D5
`[RISK][GAP]`): the quarantine-vs-recall interaction was **disproven in code,
not merely argued away** — the crystalium scout traced the actual recall
filter chain (`retrieve.py:406-428`) and confirmed post-fusion filtering is
exactly `_scope_matches` + `_is_active` (status/temporal only); there has
never been a `validation_state` filter in recall, so quarantined crystals
always surfaced. The real gap the canary exposed was different: the nexus's
handoff envelope composed a **static** `objective` string, while the canary's
distinctive per-run token lived only in the (non-FTS-indexed) brief body —
`context_handoff.sh` had to fold the task-state head into the envelope
`objective` so the token became findable (`scout-crystalium-1.8.md`
`[GAP-NEXUS-OBJECTIVE]`; fixed in nexus commit `ab22bf8` above).

## Notable campaign facts and workarounds

- **PreCompact hook refuted, not confirmed.** The spec's original T3 host row
  assumed a PreCompact hook surface on Claude Code; live verification against
  first-party docs found it **not** in the documented hook-event set
  (`evidence-host-facts.md` C1, UNVERIFIED). The design pivoted to
  **amber-zone-eager externalization** — policy rules P5/P6 checkpoint at the
  amber zone boundary specifically *because* there is no confirmed
  pre-compaction hook to anchor on; SessionStart's `compact` matcher instead
  handles post-loss pin re-injection (`evidence-host-facts.md` corrections
  §1–2).
- **FORGE's D5 quarantine risk was disproven in code, not just deliberated
  away.** See "End-to-end canary" above — `contains_tool_origin` briefs
  landing T3-quarantined never stopped them from surfacing on default recall,
  because recall never filtered on `validation_state` at any point in the
  range examined.
- **Tonberry MCP container writes were SELinux-blocked this session.** The
  workaround was an orchestrator-scribe split: the orchestrator performed
  read-side verification (drift fence comparisons, criteria SHA checks) as an
  oracle, while this chronicle hop stayed entirely on read-only tonberry
  status/list-class calls plus this document's write. No mutating
  `mcp__tonberry__*` call was made in this session.
- **Two model-safeguard false positives fired on agent resume.** These were
  worked around by dispatching a fresh-transcript maker rather than resuming
  the flagged session — a process note for future agent-resume flows in this
  campaign class.
